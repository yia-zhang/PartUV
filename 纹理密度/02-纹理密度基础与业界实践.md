# 02 · 纹理密度基础与业界实践

> 综合 leader 给的四个参考链接（RebusFarm / Beyond Extent / Foundry Modo / Polycount）
> 与对主流工具链（xatlas、Blender、UVPackmaster、RizomUV、Maya/Max、Substance、UE/Unity）
> 的系统调研。目的：把"纹理密度"这个美术概念转成研究可用的定义、数字和边界条件，
> 并划清**工程上已解决**与**研究空白**的界线。

---

## 1. 定义与公式（四个来源口径一致）

- **Texel density（TD，纹理密度）**：3D 模型表面单位真实尺寸对应的纹理像素数，
  单位 **px/cm 或 px/m**（1024 px/m = 10.24 px/cm）。
- 基本公式（RebusFarm）：`TD = Texture Resolution ÷ Object World Size`。
  例：1024² 贴图铺满 1×1 m → 10.24 px/cm。
- Modo 文档给出含 UV 的完整换算：`TD (px/m) = 贴图分辨率 (px) × UV 跨度 / 真实尺寸 (m)`，
  即 **TD 由三个量共同决定：目标密度、贴图分辨率、UV 缩放**——分辨率是显式输入，
  同样的视觉密度，贴图越大所需 UV 面积越小。
- 逐面片版本（我们研究用的定义，与 tdopt 一致）：
  `TD(f) = res · sqrt(A_uv(f) / A_3d(f))`。

## 2. 业界的标准数值（可直接用作实验参数/验收锚点）

| 数值 | 出处 | 语境 |
|---|---|---|
| 5.12 px/cm (512 px/m) | RebusFarm | background props |
| 10.24 px/cm (1024 px/m) | RebusFarm | hero assets |
| 20.48 px/cm | Beyond Extent | 个人/作品集项目，"more than enough" |
| 面部 = 身体 2–3 倍 | RebusFarm | 刻意打破均匀的行业惯例 |
| **±25%** | Beyond Extent | trimsheet 工作流允许的 TD 偏差容忍度 |
| 21 vs 28（≈25% 差） | Polycount 实例 | 两位资深美术认为"目测可接受" |
| "数倍"（several times） | Polycount (gnoop) | 密度差要到数倍量级才真正需要担心 |
| 76% | Polycount 实例 | 被认为可接受的 UV 填充率 |

**对研究的直接启示**：
1. **感知容忍度 ≈ ±25%**——与 PartUV 的失真阈值 τ=1.25 惊人地一致
   （τ 界住的 chart 内面积拉伸恰好是同一量级）。这给了"chart 内密度靠 τ 兜底、
   chart 间密度靠分配控制"的方案一个业界感知依据。
2. 均匀不是教条：**行业本来就系统性地打破均匀**（面部 2–3×、第一人称武器、
   可检视物件↑；远景 vista、被遮挡面↓）——即 L2 内容/语义感知密度不是学术臆造，
   而是**美术手工在做、没人自动化**的现有实践。
3. RebusFarm 的 **screen relation method**（按物体屏幕占比分配 texel 预算）
   与 UE 纹理 streaming 的推导逻辑同源，可作为密度需求的另一类信号。

## 3. 一致性为什么重要（以及打破它的合法理由）

- **视觉一致性**：同场景两资产密度差异过大 → 一个过锐一个模糊，破坏沉浸感
  （RebusFarm 原文）；这正是附图 GOOD/BAD 棋盘格对比展示的问题。
- **生产规范驱动**：Polycount 楼主的现实动机——freelance 客户**强制**跨资产统一
  scale/贴图尺寸/TD；Beyond Extent 作者经手的多个内部引擎**禁止提交 scale 过的物体**
  （防止隐性破坏全局 TD）。→ 一致性常是**管线级外生约束**，不是单资产美学选择。
- **合法打破**：按"对玩家的重要性 + 可接近距离"分配（Beyond Extent 原文：
  "scale up or down UV islands depending on how much of the assets will be visible
  or the importance of that part"）——**这句就是 per-part importance-weighted TD
  的业界原生表述**，我们的 L2/L3 目标有直接的实践对应物。

## 4. 工具链现状：每一类操作到底解决没解决

| 能力 | 现状 | 代表工具 |
|---|---|---|
| **测量** (measure/get) | ✅ 成熟 | Maya UV Toolkit Get、TexTools Get Density、Texel Density Checker、RizomUV pick TD + histogram、UE view modes |
| **设为均匀** (set/normalize) | ✅ 成熟 | Blender Average Islands Scale、Modo Apply Individually/Normalize、UVPM3 Set TD (px/m) + Normalize Scale、xatlas texelsPerUnit（全局单值） |
| **跨物体匹配** (match/copy) | ✅ 成熟 | Modo Sample→Apply、Maya Get→Set、TD Checker Copy TD、UVPM3 per-group TD 覆盖 |
| **锁定/保持** | ✅ 成熟 | UVPM3 Fixed Scale、Blender Lock Pinned Islands |
| **手动非均匀** | ✅ 存在但纯手工 | UVPM3 per-island Scale Multiplier（人填百分比）、ZBrush UV Master Control Painting Density（人手绘区域 ×1–4）、RizomUV Mix Scales（自动但只为省空间，与内容无关） |
| **自动内容感知分配** | ❌ **全行业缺位** | ——没有任何工具根据纹理内容/几何细节/显著性自动给 chart 分配差异化密度 |

几个值得记住的细节：
- **xatlas**：`PackOptions.texelsPerUnit` 是全局单值，**没有 per-chart 权重接口**；
  这意味着我们之前基于 xatlas 的 demand 缩放（pack2.py）已经是在做工具链没有的事。
- **Modo 的操作语义**（Sample / Apply Individually / Apply Overall / Normalize）
  是 per-island **均匀缩放、只对齐 island 平均密度**——工具明确承认 island 内部
  密度是分布而非常数（与我们"chart 内靠参数化质量、chart 间靠缩放"的分解一致）。
- **Substance Auto-Unwrap** 官方 Known limitations 原文承认做不到均匀 texel ratio
  ——自动展开工具连 L1 都不保证。
- **UE 纹理 streaming** 在运行时按屏幕占比自动推导"每世界单位需要多少 texel"
  并用红/白/绿 view mode 可视化，但**这个信息从未反馈回 UV 生成阶段**——
  "需求推导已存在、分配执行缺位"的又一例证。

## 5. 划界结论：工程基线 vs 研究空白

> **已解决**：measure、set-uniform、match、lock、手动倍率——即"执行层"齐备
> （UVPM3 的 per-island iparam、xatlas 源码改造都可以执行任意给定的分配）。
> **缺位**：**决策层**——从信号（纹理内容频率、几何细节、显著性、预期视距）
> 自动映射到 per-chart/per-part 密度预算，并在打包中带约束地执行。
> 这正是本课题（04 文档档 A/B/C）要填的位置；需求信号侧则由 TDF 项目补上
> "从参考图预测"这一最缺的输入。

另外两条来自实践者的松弛条件，对算法设计很有价值：

1. **不必追求严格等密度**——±25% 以内感知无碍（Polycount + Beyond Extent 双重印证），
   算法可以把这 25% 的松弛全部让给 packing 效率；
2. **密度目标常是场景/图集级的全局规范**（客户/引擎强制），所以接口上
   "全局目标 TD + per-part 倍率"比"每 part 独立绝对值"更贴近生产习惯
   （whitemodel.py 的 `--td` 绝对锚定 + 推荐分辨率的设计方向是对的）。
