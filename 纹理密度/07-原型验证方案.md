## 1. 两段式理解与既有分析的对应关系（自洽，但串联方式有讲究）

| 你的表述 | 对应既有分析 | 角色 |
|---|---|---|
| (1) 白模+image → 带纹理物体 | 01 文档管线第③段（纹理生成，用现成开源/Meshy 管线） | 价值兑现场，不重造 |
| (2) 带纹理物体 → 密度感知新 UV → 重新贴图 | 03 文档的 A→B relayout + 04 文档档 A（底座换成 PartUV） | **本课题主体** |
| (1)+(2) 串联 | 01/04 文档的**两遍法**（已确认决策的需求信号路径之一） | 端到端故事 |

**串联方式有两种，价值不同（源自 03 文档 §2.4 的信息论边界）**：

- **串联 α（原型版）**：(1) 均匀布局生成 → (2) 重排 + **rebake** → 完成。
  快、够原型用；但 rebake 是重采样，第一遍生成时某 part 若已欠采样（糊），
  第二遍给它再多 UV 面积也只是放大模糊像素。
- **串联 β（完整版）**：(1) 粗生成 → (2) 重排 → **回到 (1) 在新布局上重新
  烘焙/重新生成** → 完成。第二遍是重新生成而非重采样，**无信息损失**，
  这才是密度控制的完整价值（代价 ≈ 2× 生成开销）。

**原型阶段规避信息损失的实验技巧**：oracle 用高分辨率纹理（4K 原图或 4K 生成
结果）→ 分别 rebake 到 **1K 新布局 vs 1K 旧布局** → 比渲染质量（LPIPS/锐度）。
高分辨率源保证了"信息充足"，两个 1K 结果的差异就干净地度量了**布局本身**的收益。
这应该作为档 A 的核心定量实验。

## 2. "end-to-end 模型"的两种含义（向 leader 汇报时建议区分）

- **(a) 端到端管线**（pipeline，各模块是独立算法/模型）：原型验证做的就是它，
  短期完全可达；
- **(b) 端到端可微/联合训练的单一模型**：远期开放问题——可微打包是离散组合
  难题，学术上只有部分参考（Knodt TOG 2024 的可微 UV 优化联合了参数化与烘焙，
  但没有碰打包与 part 结构）。

建议对 leader 的措辞用"**端到端管线原型**"，不要现在承诺 (b)——它本身可以是
课题的远期第二篇故事，而不是第一步的验收标准。

## 3. 基于开源 PartUV 的可行性：本机冒烟测试实录（2026-07-13）

结论：**可行，且落地成本比预想低**。逐项事实：

1. **环境现成**：本机已有 `partfield` conda 环境（Python 3.10, torch 2.4+cu124,
   CUDA 可用），`pip install partuv`（预编译 wheel）直接成功；
2. **checkpoint 已在本地**：`/root/zhaotianhao/PartField/model/model_objaverse.ckpt`
   （无需下载）；
3. **API 基本正合需求，但有一个语义坑（已踩明白）**：
   ```python
   from partuv.preprocess_utils.partfield_official.run_PF import PFInferenceModel
   import partuv
   pf = PFInferenceModel(device="cuda", checkpoint_path=CKPT)
   mesh, tree_file, tree_dict, _ = partuv.preprocess(mesh_path, pf, out_dir,
                                       merge_vertices_epsilon=None)   # 官方 demo 同款
   final, parts = partuv.pipeline_numpy(V=mesh.vertices, F=mesh.faces,
                                        tree_dict=tree_dict, configPath=CFG,
                                        threshold=1.25)
   # ⚠ final(final_parts) 是所有 part 合并后的单一 buffer(供导出 OBJ 用),
   #   其 .faces 是 per-part 局部索引的拼接——不能当全局面索引用!
   # ✅ 真正的 chart 结构在 parts[i].components(每个 part 的 charts),
   #   每个 Component 有 .V .F .UV .distortion(3D 顶点保留原坐标)
   ```
4. **全局面对应的正确做法（已验证）**：用 **面质心匹配**（chart 的 3D 面质心
   ↔ 输入 mesh 面质心，cKDTree 精确查询）。实测两个真实 objaverse mesh 上
   **匹配 100%、零误差**（d < 1e-8）——per-face 内容权重与 rebake 需要的
   对应关系稳定可得，且对任何顶点重排都鲁棒。
5. **冒烟运行结果**：见文末附录（复跑脚本已归档到本目录 `实验/`）。

**两个集成注意事项（原型时要处理）**：

- `preprocess` 会**剥离输入 mesh 的已有 UV**、并做 merge/fix（官方 demo 用
  `merge_vertices_epsilon=None` 关掉顶点合并）。所以"(2) 的输入是带纹理 mesh"
  的正确做法：**自己先把原 mesh 的 (UV, texture) 存下来**，PartUV 只喂几何；
  之后按面索引对应回去算 per-face 内容权重与 rebake。
- 若 `fix_mesh_trimesh` 改变了面数（坏拓扑输入时可能），面索引对应会断——
  兜底方案是 3D 最近点对应（`trimesh.proximity`：新 UV texel → 3D 点 →
  原 mesh 最近表面点 → 原 UV → 取色），对任何重网格都稳。冒烟脚本里
  对此做了显式检查。

   密度层的真正留白是**显式控制与内容感知分配（L2/L3）**——config 无 density
   参数、无 per-chart 权重接口。布局的构建依据（官方 blender 路径）三层：
   chart 形状=ABF 保角展开；相对大小=纯 3D 面积等比（与内容无关）；

   （统一全局缩放，这就是"每张图都填满 UV 空间"的原因）。
2. **⚠ τ 不是硬保证**：该 mesh 实测单 chart 面积失真最大 **1.661 > τ=1.25**。
   论文搜索期用简化网格的代理失真估计，最终全分辨率 ABF 可能超界（小 chart /
   坏局部尤甚）。对我们的含义：**L1 层不能盲信 τ，密度层必须按每个 chart 的
   实测平均尺度来分配**（我们本来就是这么算的），τ 只作为 chart 内均匀性的
   近似先验；论文写作时这也是一个可报告的观察。
3a. **⚠⚠ 双面重合面片（twin faces）会击穿最近邻质心匹配**（2026-07-14，
   复杂资产 `8d1b6fc…glb` 实测：10 材质 / 774 组件 / **28.9% 面为顶点重合的双面几何**）：
   kd 最近邻把孪生面折叠到同一索引 → coverage 虚报 5.79% 丢面（实际 PartUV 只真丢 2 面！）；
   更严重的是 `blender_repack` 的打包往返对应同样折叠 → 会给孪生一层取到**另一层的 UV**
   （布局损坏）。**已修复**：`FaceMatcher`（质心桶式一对一 + 候选>1 时按法线方向判别 +
   kd 兜底），应用于 notebook 三处对应逻辑；实测 coverage 0.861→1.0000，无孪生资产行为
   不变（车轮回归通过）。顺带验证：该资产的**平铺 UV（uv∈[0,2]）经 trimesh 合并内容忠实**
   （颜色误差 0.015），多材质/分辨率悬殊/纯色材质均被正确合并——多纹理复杂资产的
   主要风险清单至此闭环。
3. **⚠ PartUV 会静默丢面**（2026-07-13 二次诊断，8 网格扫描）：拓扑碎的 AI
   网格上，管线会把少量输入面排除在所有 chart 之外——实测丢的几乎全是
   **孤立单三角形/1–4 面的小碎片连通组件**（论文 3.5 节多组件 fallback 的行为），
   例：`objaverse_1389004d` 624 面丢 33（其中 32 个是独立单面组件），
   `objaverse_7b3232` 2454 面丢 28；面积占比均 <2%。质心匹配本身在全部 8 个
   网格上零误差、无重复质心。notebook 已改为：只对真正的质心失配硬报错，
   丢弃面如实报告并从统计中排除（热力图画灰色）。**含义**：密度层的输入约定
   应为"chart 覆盖的面集合"而非"全部输入面"；丢弃面占比可作为网格质量预检指标。
4. **⚠ 逐面 CV 是有陷阱的指标**：`objaverse_7b3232` 原始 UV 的逐面 TD CV=2.18，
   但**面积加权 CV 仅 0.078**——不均匀全部来自微小三角形，感知上它本来就均匀
